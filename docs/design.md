# 盒记 UI 设计规范

> 适用范围：盒记 Android 本地记账 App。上游参考为 Vercel Geist 的 token / 组件状态思想；本文件只保留盒记可直接执行的规则。
>
> 写或改 UI 前，先读本文件、`src/uni.scss` 和目标页面的现有组件。业务规则仍以 `AGENTS.md`、`memory-bank/design-document.md` 为准。

## 1. 产品语境与页面任务

盒记服务于盒饭档口经营者的日常闭环：录单、配送、当晚对账。

- **今日页**：保留既有的今日概览、状态计数与订单分组；本轮不改变配送流程。
- **订单表单**：按“对象 → 份数 → 支付 → 金额 → 确认”完成一次可预期的录单或修改。
- **统计页**：帮助对账，而不是展示抽象经营指标；优先呈现收入、支出、利润及它们随日期的变化。

本轮的标志性结构是**操作摘要**：在新建和编辑订单页，把本次即将保存的金额放在一个明确区域，其余信息保持安静、可扫读。

## 2. Token

视觉值定义在 `src/uni.scss`；新增或修改的业务页面优先引用下列语义 token，禁止临时发明颜色、圆角和阴影。

| 类别 | token | 用途 |
|---|---|---|
| 画布 | `$hej-color-canvas` | 页面背景 |
| 表面 | `$hej-color-surface` / `$hej-color-surface-subtle` | 卡片、弱分区、固定确认区 |
| 文字 | `$hej-color-text` / `$hej-color-text-secondary` / `$hej-color-text-tertiary` | 标题、说明、辅助信息 |
| 边界 | `$hej-color-border` | 卡片、输入和分隔线 |
| 主动作 | `$hej-color-accent` / `$hej-color-accent-soft` | 新建、保存、待配送状态 |
| 成功 | `$hej-color-success` / `$hej-color-success-soft` | 已配送、正利润 |
| 提醒 | `$hej-color-warning` / `$hej-color-warning-soft` | 取消、待注意状态 |
| 危险 | `$hej-color-danger` / `$hej-color-danger-soft` | 删除、不可逆错误 |
| 布局 | `$hej-space-*` / `$hej-radius-*` / `$hej-shadow-panel` | 间距、圆角、卡片层次 |

使用 `$hej-*` 的 Vue 样式块必须声明 `<style ... lang="scss">`；否则 token 会作为无效 CSS 原样输出，App 会回退到平台默认样式。

当前 App 是**浅色优先**，尚未提供可切换的暗色运行时主题；不得把“暗色支持”写成已实现。后续引入暗色时，必须保持上述语义 token 名不变，只替换值并重新验收可读性。

## 3. 组件与状态规则

### 按钮

- 主按钮只用于当前页面的唯一关键提交：如“保存订单”“标记已配送”。使用实色 `$hej-color-accent`。
- 次按钮用于可逆或辅助动作：如“取消编辑”“查看原订单”。使用弱表面或描边，不与主按钮争抢颜色。
- 危险按钮只用于取消、删除等不可逆动作；删除应与常规操作分区。
- `disabled` 必须降低对比度并禁止重复提交；`loading` 文案说明正在执行的动作，如“保存中…”。
- 自定义 `<button>` 必须显式设置 `height` 与等值 `line-height`，并设置 `text-align: center`，避免 App 端文字偏移。
- H5 端新增可点击元素需有清晰 hover / active / focus-visible；App 端至少保留明确的 active 反馈，主操作触达高度不小于 88rpx。

### 状态

- 待配送：主动作蓝色，始终优先于历史状态。
- 已配送：成功绿色，只作为完成反馈，不抢主操作视觉。
- 已取消：提醒橙色；删除错误才使用危险红色。
- 状态名称、提示和按钮用同一套动词：例如“标记已配送”成功后提示“已标记配送”。

### 空态

空态必须回答三件事：当前没有什么、这意味着什么、下一步能做什么。

- 不使用只有“暂无数据”的孤立空态。
- 有直接下一步时，提供一个主动作按钮；订单列表空态应能创建当前所选日期的首单。
- 加载态与空态分开，加载时不显示误导性的“暂无”。

### 卡片、列表与表单

- 卡片用于承载一个完整判断或一组相关字段；不要为每一行内容再包一层卡片。
- 页面标题和区块标题说明当前任务；数字与金额使用高对比、稳定的字级，不用渐变或装饰性大字。
- 表单按业务决策顺序分区：订单对象、配送与支付（含金额）、备注。金额由支付方式决定，放在配送与支付卡内，以分隔线标出小节，不另起卡片；校验和上下文提示紧贴相关分区。
- 新建和编辑订单的确认区固定在屏幕底部，只显示本次实际金额、合并提示（如有）和确认动作。
- 动效仅用于折叠、加载、保存等状态变化；不新增装饰性动画。

## 4. 内容规则

- 用用户能识别的业务词：`入账收入`、`有效订单`、`本次实际金额`、`收支 / 利润趋势`。
- 支付份数必须写出实际渠道：`1 份微信支付`、`1 份现金支付`；组合支付同时写次卡次数与微信 / 现金份数，不使用笼统的“货币支付”。
- 按钮使用“动词 + 对象”：`新建订单`、`保存订单`、`标记已配送`，避免泛化的“确定”“提交”。
- 错误提示说明原因和可行的下一步；已有业务错误文案优先保留，不用纯视觉改动掩盖它。

## 5. UI 验收清单

每次改 UI 后逐项确认：

- [ ] 新增样式复用了本文件和 `src/uni.scss` 的 token；没有无理由的硬编码颜色、圆角、阴影。
- [ ] 使用 `$hej-*` 的样式块均声明 `lang="scss"`，`pnpm test` 的样式预处理检查通过。
- [ ] 当前页面只有一个视觉上的主动作；危险动作和常规动作有明确分区。
- [ ] 自定义按钮的高度、行高和文字对齐已成对设置，App 端文字垂直居中。
- [ ] loading、empty、error、disabled、保存中状态都有可读文案。
- [ ] 空态提供与当前上下文一致的下一步。
- [ ] 状态颜色不承担唯一信息，文字仍能表达状态。
- [ ] 手机宽度下文本不截断关键金额、订单数量或确认按钮；主操作触达区域至少 88rpx。
- [ ] 表单底部有足够滚动留白，固定确认区不遮挡最后一个字段。
- [ ] 不改动金额计算、订单状态机、SQLite 写入与既有业务错误边界。
- [ ] 运行 `pnpm test`、`pnpm type-check`、`pnpm lint`、`pnpm build:h5`、`git diff --check`；Android 交互仍以 HBuilderX 真机回归为准。

## 6. 例外

`src/pages.json`、uni-ui 第三方组件和平台原生配置可能必须使用静态色值；这类值应与 token 对齐，并在代码评审时说明原因。现有未触及页面的历史样式不要求在一次需求中全量重写，但新的视觉改动不得继续扩大旧的硬编码模式。
