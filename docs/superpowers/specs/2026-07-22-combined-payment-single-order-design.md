# 组合支付、次卡预占与一餐一单设计

> 状态：用户已复核并完成实现；CLI / H5 验证通过，HBuilderX 真机回归待执行
> 日期：2026-07-22

## 1. 背景与根因

当前订单只能保存一种 `payment_method`。纯次卡订单的 `amount=0`，配送时按整张订单的 `quantity` 从客户次卡余额池扣次。

现有“次卡剩余 1 次但新增 2 份次卡订单仍能保存”的根因有两层：

1. `src/pages/order/new.vue` 只要能找到任意一张剩余次数大于 0 的 active 次卡，就允许保存；没有比较订单份数与客户总可用次数。
2. `src/api/orders.ts` 的创建流程只校验参考次卡存在、属于当前客户且状态为 active；余额校验被旧设计明确推迟到 `markDelivered()`。

因此只在页面补一个判断不能解决根因。新设计必须同时改变订单支付模型、API 写入约束和待配送次数的预占口径。

## 2. 已确认的产品规则

### 2.1 一餐一单

- 同一 `customer_id + order_date + meal_type` 只能有一张有效订单。
- 再次新增时，新增页填写的是“本次增量”，保存后合入已有 pending 订单。
- 合并保留已有订单 ID 和 `sort_order`，不自动置顶或沉底。
- 已有 delivered 订单时禁止继续新增，并提示该客户本餐次已配送。
- 只有 cancelled 历史订单时允许创建新订单，不复活旧订单。
- 编辑订单改变客户、日期或餐次后若撞到另一张 pending 订单，必须二次确认后合并；目标已配送时拒绝修改。

### 2.2 允许的支付组合

允许：

- 纯微信
- 纯现金
- 纯次卡
- 次卡 + 微信
- 次卡 + 现金

不允许：

- 微信 + 现金
- 次卡 + 微信 + 现金
- 一张订单内存在多个货币单价批次

组合支付时，次卡次数初始为空，由用户手动填写。系统不默认尽量使用次卡，也不静默切换支付方式。

### 2.3 价格与备注

- 货币支付份数 = `quantity - meal_card_quantity`。
- 货币金额 = 货币支付份数 × 实际单价，金额运算必须走 big.js helper。
- 一张订单只有一个实际单价。
- 重复新增时带出已有实际单价；若用户输入的新单价不同，提交前弹窗展示旧单价、新单价、受影响份数和重算后金额。
- 用户确认后，新单价应用于整张订单的全部微信或现金份数。
- 新备注为空时保留旧备注；旧备注为空时采用新备注；两个非空备注不同则用 `；` 按录入顺序拼接；trim 后完全相同则不重复。

### 2.4 整单流转

- 合并订单只能整单配送、取消或删除，不支持部分状态流转。
- 配送时只扣 `meal_card_quantity`，货币部分不参与扣次。
- 删除已配送组合订单时，按 `meal_card_usages` 精确回滚其中的次卡使用明细。
- 配送时若余额意外不足，整个事务回滚，不做部分扣次，也不自动修改支付方式；页面提示所需次数和当前可用次数，并提供“去编辑支付”入口。

## 3. 方案比较

### 方案 A：拆成多张订单，列表临时聚合

优点是能复用现有单支付订单字段。缺点是一名客户的一次配送会产生多个订单 ID、多个状态和多个编辑入口，取消与删除也容易只处理一部分。该方案与“一名客户一个餐次只配送一次”的业务事实冲突，淘汰。

### 方案 B：新增通用 `order_payments` 子表

每种支付方式写一条支付明细，扩展性最好，未来可以支持微信 + 现金或三种混用。但本次已经明确只允许“次卡 + 一种货币支付”，通用支付表会增加新表、关联查询、备份顺序和编辑聚合成本，属于超出当前需求的抽象。

### 方案 C：在 `orders` 增加 `meal_card_quantity`（推荐）

继续用一行 `orders` 表示一张逻辑订单，只增加明确的次卡支付次数：

- `quantity`：整单总份数。
- `meal_card_quantity`：本单承诺使用的次卡次数。
- `payment_method`：纯次卡时为 `meal_card`；有货币部分时为 `wechat` 或 `cash`，表示唯一的补款渠道。
- `unit_price`：货币部分的实际单价；纯次卡订单继续保留参考单价，优先取用户输入或客户默认价 × 折扣率，两者都缺失时才回退最早可用次卡的次均价。
- `amount`：仅记录微信或现金收入；纯次卡为 0。
- `meal_card_id`：仍只作为参考卡 / 首张实际扣次卡，精确扣次以 `meal_card_usages` 为准。

该方案完整覆盖已确认的支付范围，不增加依赖或通用框架，也能保持现有统计口径：订单收入继续汇总 `orders.amount`，开卡收入继续汇总 `meal_cards.amount`。

## 4. 数据约束

### 4.1 支付形态不变量

| 形态 | `payment_method` | `meal_card_quantity` | `amount` |
|---|---|---:|---:|
| 纯微信 | `wechat` | 0 | `unit_price × quantity` |
| 纯现金 | `cash` | 0 | `unit_price × quantity` |
| 纯次卡 | `meal_card` | `quantity` | 0 |
| 次卡 + 微信 | `wechat` | `1..quantity-1` | `unit_price × (quantity - meal_card_quantity)` |
| 次卡 + 现金 | `cash` | `1..quantity-1` | `unit_price × (quantity - meal_card_quantity)` |

API 必须拒绝负数、非整数、超过订单份数的次卡次数，以及与支付方式不一致的字段组合。组合支付在总份数为 1 时不可选择。

### 4.2 次卡余额与预占

创建订单不修改 `meal_cards.used_meals`，仍然只在配送时真正扣次。但所有 pending 订单的 `meal_card_quantity` 都会占用“可承诺次数”。

计算口径：

```text
客户实际剩余 = Σ(active 卡的 total_meals - used_meals)
其他待配送预占 = Σ(其他 pending 订单的 meal_card_quantity)
当前订单可用 = 客户实际剩余 - 其他待配送预占
```

- 新建订单校验本次或合并后的次卡次数不超过当前可用。
- 编辑订单时从“其他待配送预占”中排除当前订单。
- 合并到已有订单时排除目标订单，再校验合并后的总次卡次数。
- 取消或删除 pending 订单后，无需写额外释放记录；它自然不再进入预占查询。
- UI 展示“实际剩余 / 其他订单已预占 / 当前可用”，但 API 在事务内重新查询并作为最终裁决。

### 4.3 充值记录校正联动

修改单张次卡的 `total_meals` 时继续保证 `total_meals >= used_meals`，并增加客户余额池校验：修改后的客户总剩余次数不得小于该客户所有 pending 订单的预占总数。多张 active 卡仍按客户余额池计算，不把全部预占错误地绑定到某一张卡。

## 5. 创建与合并流程

### 5.1 新增页识别已有订单

客户、日期或餐次变化时查询同键非 cancelled 订单：

- 无有效订单：按普通新建处理。
- 有 pending 订单：页面展示已有份数、支付摘要和“本次将合并到已有订单”；表单份数仍代表本次增量。
- 有 delivered 订单：显示不可新增状态，提交时 API 再次拒绝。

### 5.2 保存增量

API 在一个事务中完成：

1. 重新读取同键有效订单和次卡可用次数。
2. 校验本次支付拆分。
3. 无 pending 订单时插入新订单并追加到当前餐次末尾。
4. 有 pending 订单时校验支付兼容性，累加总份数与次卡次数，合并备注，并保留原 ID / 排序。
5. 若新旧订单都含货币部分且支付渠道不同，抛出支付冲突错误，不修改任何数据。
6. 若货币渠道相同但实际单价不同，先返回需要确认的新旧价格与重算预览；用户确认后重试并更新整张订单。
7. 返回最终唯一订单。

同一设备虽然并发概率低，仍以事务内二次查询为准，避免双击或页面缓存产生重复订单。

## 6. 编辑与状态流程

### 6.1 编辑当前订单

- 编辑页中的 `quantity` 是整张订单总份数，不是增量。
- 已有组合支付的 `meal_card_quantity` 原样带出。
- 修改总份数时不自动调整次卡次数；货币份数与金额随总份数重算。
- 若组合支付中的次卡次数已经不小于总份数，表单与 API 都拒绝保存，要求用户手动调整或切换纯次卡。
- 修改客户、日期或餐次后若命中另一张 pending 订单，弹出合并确认；确认后保留目标订单 ID / 排序并删除当前订单。

### 6.2 配送、取消与删除

- `markDelivered()` 在事务内按 `meal_card_quantity` 从客户 active 卡池 FIFO 扣次并写 `meal_card_usages`。
- 任何一张卡或任意一步写入失败都回滚订单状态、排序、扣次和 usage。
- 取消 pending 订单只改状态；预占由查询口径自然释放。
- 删除 delivered 且 `meal_card_quantity > 0` 的订单时按 usage 回滚；旧数据无 usage 时才走兼容回退。
- 原“次卡不足后整单改微信 / 现金并立即配送”的快捷分支废除，改为进入订单编辑页重新分配。

## 7. 页面设计

### 7.1 新增 / 编辑表单

两页涉及的业务字段统一由 `<uni-forms>` + `<uni-forms-item>` 承载并通过 `formRef.validate()` 提交。

一级支付选项：

```text
微信 | 现金 | 次卡 | 组合支付
```

组合支付展开：

- 补款方式：微信 / 现金。
- 次卡次数：必填，进入组合支付时初始为空；编辑已有组合订单时带出已保存值。
- 货币份数：自动显示。
- 实际单价：默认客户餐次价 × 折扣率，可手动修改。
- 补款金额：自动计算。
- 次卡提示：实际剩余、其他订单已预占、当前可用。

纯次卡数量超过可用次数时立即显示行内警告；保存按钮不因这一项静默禁用，点击后仍触发表单与 API 错误提示：

> 当前可用 1 次，订单需要 2 次，请减少份数或改用组合支付。

### 7.2 订单列表

保留按日期筛选、午餐 / 晚餐折叠、拖拽排序和状态标签。由于数据层已经保证“一餐一单”，列表不再另做临时聚合。

删除列表项最右侧的“次卡 / 价格”区域，副标题固定按以下顺序拼接：

```text
餐次 · 总份数 · 次卡次数 · 微信/现金总金额 · 备注
```

示例：

```text
午餐 · 2份 · 次卡 1次 · 微信 ¥25.00 · 不要葱
```

不存在的支付部分和空备注直接省略。备注允许自然换行并完整显示，不使用省略号截断。

### 7.3 订单详情

- 只读态分别展示总份数、次卡次数、货币渠道、实际单价和货币总金额。
- 支付摘要使用“次卡 1次 + 微信 ¥25.00”等组合文案。
- pending 订单可编辑；delivered / cancelled 保持只读。
- 配送余额不足时弹出明确错误与“去编辑支付”入口，不自动改价或改支付。

## 8. Schema、迁移与备份

### 8.1 Schema v5

`orders` 新增：

```sql
meal_card_quantity INTEGER NOT NULL DEFAULT 0
```

新库 DDL 直接包含该字段；`MIGRATIONS` 末尾追加 v5 段，不修改 v1-v4：

1. `ALTER TABLE orders ADD COLUMN meal_card_quantity ...`。
2. 旧 `payment_method='meal_card'` 订单回填 `meal_card_quantity=quantity`。
3. 其他旧订单回填 0。

### 8.2 历史重复 pending 订单

升级时只自动合并支付渠道和单价都兼容的 pending 重复订单：保留最早订单 ID 与既有排序，累加份数和次卡次数，按已确认规则合并备注。

以下历史数据不自动改写，只记录可读诊断并保留：

- delivered 或 cancelled 重复历史。
- 微信与现金冲突的 pending 重复订单。
- 无法无损统一实际单价的 pending 重复订单。
- pending 与 delivered 混合的异常组合。

新版本 API 不再产生新的重复有效订单。由于保守保留的历史异常可能违反唯一性，不新增会让升级失败的数据库唯一索引，以事务内业务校验维护新写入不变量。

### 8.3 备份兼容

- 新备份写入 schema v5，并随 `orders` 导出 `meal_card_quantity`。
- 导入 v1-v4 时按旧 `payment_method` 推导：次卡订单等于 `quantity`，微信 / 现金订单为 0。
- 导入后执行与数据库升级相同的“仅合并兼容 pending 重复订单”规则。
- `meal_card_usages` 仍是已配送扣次与回滚的事实来源，导入顺序继续保证外键安全。

## 9. 错误处理与事务边界

新增或细化可辨识的业务错误：

- 次卡需要次数与当前可用次数不匹配。
- 同键订单已经配送。
- 合并时微信 / 现金渠道冲突。
- 合并价格变化，需要用户确认。
- 编辑目标存在 pending 订单，需要用户确认合并。
- 修改充值记录会侵占 pending 订单预占次数。

页面只捕获并翻译这些业务错误；未知数据库错误统一提示保存失败并保持原数据。创建、合并、编辑冲突合并、配送、删除回滚与充值记录校正全部在 `tx()` 中完成。

## 11. 验证方案

### 11.1 缺陷复现与自动化 / 脚本验证

实现前保留最小失败用例：客户实际剩 1 次，新增 2 份纯次卡订单，旧代码允许创建。修复后同一用例必须在 API 层拒绝，并验证下面的预占用例：

1. 实际剩 2 次，先建 1 次 pending，再建需要 2 次的订单时拒绝。
2. 编辑当前 pending 时排除自身预占，原值可保存。
3. 取消 pending 后，预占次数立即可再次使用。
4. 组合支付 2 份、次卡 1 次、微信单价 ¥25，保存金额为 ¥25，配送只扣 1 次。
5. 删除上述 delivered 订单后，只回滚 1 次。
6. 微信订单再次新增现金时拒绝，不生成第二张订单。
7. 同支付重复新增合并到原 ID，排序不变，备注正确拼接。
8. 合并时单价变化，未确认不写入；确认后按新单价重算整单货币部分。

### 11.2 静态门禁

- `pnpm type-check`
- `pnpm lint`
- `pnpm build:h5`

### 11.3 HBuilderX 真机回归

- 新建 / 编辑四种一级支付形态。
- 组合支付次卡次数初始为空、非法范围提示正确。
- 新增页识别并合并已有 pending 订单，价格变化弹窗内容准确。
- 纯次卡余额不足与 pending 重复预占均在保存前被拦截。
- 组合订单配送、取消、删除和 usage 回滚正确。
- 订单列表副标题顺序、金额、次数和长备注换行正确。
- v4 真机数据库自动升级 v5；v1-v4 备份可恢复，v5 备份可往返。
- Dashboard、统计页收入仍只计货币订单金额 + 开卡收入，取消订单仍不计入。

## 12. 验收标准

- 一名客户在同一日期同一餐次只出现一张有效订单，并只配送一次。
- 重复新增可按增量无损合并，编辑冲突有明确确认，不会静默改支付或改单价。
- 组合支付只能由次卡和一种货币渠道构成，次卡次数必须由用户明确填写。
- 新增、编辑、充值记录校正和配送都遵守同一套次卡预占口径。
- 余额不足时创建失败且提示包含需要次数与当前可用次数，不再把错误推迟到配送才发现。
- 列表不再显示末端“次卡 / 价格”，副标题完整呈现餐次、份数、支付拆分和备注。
- schema v5、旧数据迁移、备份恢复、统计口径与文档保持一致。
